# Tagging Guide

이 문서는 노트/블로그 문서에 태그를 붙일 때 사람이든 AI든 동일한 규칙으로 작업하기 위한 기준이다.

## 목적
- 검색성과 연결성을 높인다.
- 문서 간 주제 축을 일관되게 유지한다.
- 새 태그를 허용하되, 난립을 억제한다.

## 기본 규칙
- 태그 개수: 문서당 최소 1개, 최대 3개
- 표기: kebab-case (예: `note-taking`, `side-project`)
- 대소문자: 기본 소문자, 이니셜/약어만 대문자 유지
- 허용 약어 예시: `AI`, `LLM`, `AWS`, `DNS`, `HTTP`, `HTTPS`, `API`, `SQL`, `SSR`, `CSR`, `FSD`, `CDK`, `VPC`, `ENI`, `EIP`, `XSS`, `CSRF`, `RAG`
- 한글 태그는 사용하지 않는다. 의미에 맞는 영문 태그로 변환한다.

## 태깅 우선순위
- 1순위: 기존 태그 재사용 (`tag-index.md` 우선 참조)
- 2순위: 문서 핵심 주제 1개
- 3순위: 기술/맥락 1개
- 4순위: 문서 성격(예: `tutorial`, `retrospective`, `book`) 1개
- 주제가 겹치면 더 구체적인 태그를 우선한다. (예: `web`보다 `SSR`)

## 완화 모드 신규 태그 정책
- 새 태그는 즉시 허용한다.
- 단, 아래 조건을 만족해야 한다:
  - 기존 태그로 의미를 충분히 표현하기 어렵다
  - 너무 포괄적이지 않다 (`tech`, `dev` 같은 태그 금지)
  - 너무 개인적이지 않다 (특정 일회성 맥락 지양)
- 새 태그를 만들면 `tag-index.md`와 `tag-relationships.md`를 함께 갱신한다.

## 추천 조합 패턴
- 기술문서: `기술` + `영역` + `형식`
  - 예: `react`, `frontend`, `data-fetching`
- 개념정리: `주제` + `상위영역`
  - 예: `DNS`, `network`
- 회고/에세이: `성격` + `주제`
  - 예: `retrospective`, `career`
- 책정리: `book` + `주제` (+ 선택 1개)

## AI 요청 템플릿
```text
다음 문서에 태그를 붙여줘. 규칙은 note/tagging-guide.md를 따르고, 기존 태그는 note/tag-index.md를 우선 재사용해줘.
출력 형식:
- final_tags: [최소 1개, 최대 3개]
- reused_tags: [기존 태그]
- new_tags: [신규 태그, 없으면 []]
- reason: 한두 문장으로 근거
```

## 운영 루틴
- 새 태그가 생기면 즉시 인덱스/관계 문서 갱신
- 주 1회: 유사 태그 점검 (`note-taking` vs `notes` 등)
- 월 1회: 거의 쓰이지 않는 태그 통폐합 검토
